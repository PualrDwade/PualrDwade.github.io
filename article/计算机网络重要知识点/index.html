计算机网络重要知识点

本文图片来自github的CS-NOTE,在此感谢作者CYC的开源精神

一. 互联网

1.1 基本概念

网络是多个终端(主机)互相连接起来,而互联网则是将网络联系起来,互联网是网络的网络

1.2 主机之间的常见通信方式:

  • 客户机服务器模式:(C/S),拓扑结构呈放射状

  • 点对点模式(不区分客户机和服务器,公平通信):(P2P),拓扑结构呈网状

1.3 计算机网络体系结构

数据工作流:

应用层:HTTP等协议数据--->传输层:TCP/UDP数据报,面向进程--->网络层:传输层数据报封装成组:面向主机

--->数据链路层:将网络层的数据组封装成帧:面向主机之间的链路-->物理层:传输比特流,单位为bit.

总之,底层为高层提供服务,数据从高层不断拆箱成比特流数据到另一个主机,而后反过来不断装箱,提供应用层使用,这便是网络数据的工作流,HTTP->成数据报->成组->成帧->成比特->帧->分组->数据报->HTTP

二. 物理层

物理层进行二进制的比特流传输.重点是通信方式,主要有3种通信方式

1. 单工通信:数据固定由一端传输到另一端

2. 半双工通信:数据可以双向传输,但是不可以同时在两端传递,只能交替传输

3. 全双工通信:数据可以同时在两端传递

三. 数据链路层

主要功能是报文分组之后的成帧传输,面向链路服务

3.1 成帧

将网络层传下来的报文分组添加首部和尾部,用于标记帧的开始和结束

3.2 差错检验

数据链路层广泛使用了循环冗余检验（CRC）来检查比特差错

3.3 信道

1. 广播信道:一对多通信,所有节点在同一个广播信道发送数据,因此可能产生碰撞,需要使用算法来规避.

主要有两种控制方法进行,一个是使用信道复用技术,一是使用CSMA/CD协议

2. 点对点信道:PPP协议控制,不会冲突,过程简单

3.4 信道复用技术

1. 频分复用

频分复用的所有主机在相同的时间占用不同的频率带宽资源

2. 时分复用

3.5 MAC地址

MAC 地址是链路层地址,长度为6字节(48 位),用于唯一标识网络适配器(网卡)

一台主机拥有多少个网络适配器就有多少个 MAC 地址.例如笔记本电脑普遍存在无线网络适配器和有线网络适配器,因此就有两个 MAC 地址

四. 网络层

网络层的主要作用是报文的分组和数据传输,路由的寻址算法,是整个互联网的核心

使用 IP 协议，可以把异构的物理网络连接起来，使得在网络层看起来好像是一个统一的网络

​

五. 传输层

网络层只把分组发送到目的主机，但是真正通信的并不是主机而是主机中的进程。传输层提供了进程间的逻辑通信，传输层向高层用户屏蔽了下面网络层的核心细节，使应用程序看起来像是在两个传输层实体之间有一条端到端的逻辑通信信道

传输层的对象是进程

5.1 TCP和UDP

  • 用户数据报协议 UDP（User Datagram Protocol）是无连接的，尽最大可能交付，没有拥塞控制，面向报文（对于应用程序传下来的报文不合并也不拆分，只是添加 UDP 首部），支持一对一、一对多、多对一和多对多的交互通信。
  • 传输控制协议 TCP（Transmission Control Protocol）是面向连接的，提供可靠交付，有流量控制，拥塞控制，提供全双工通信，面向字节流（把应用层传下来的报文看成字节流，把字节流组织成大小不等的数据块），每一条 TCP 连接只能是点对点的（一对一）。

六. 应用层

HTTP协议

超文本传输协议,建立于TCP之上的应用层协议

一个HTTP过程包括请求报文与响应报文两个部分

请求报文

请求头,请求方法,请求体

第一行为请求行,包含了请求方法和请求URL,常用的HTTP请求方法如下:

  • GET:获取资源

  • POST:发送资源

  • HEAD:获取报文首部,与GET相同但是不返回BODY,用于判断资源可访问性

  • PUT:上传文件,没有验证机制,不安全,也可以修改资源

  • PATCH:部分修改资源,PUT只能完全提到原始资源,而PATCH可以部分修改

  • DELETE:删除资源

  • OPTIONS:查询资源所支持的操作

    会返回 Allow: GET, POST, HEAD, OPTIONS 这样的内容。

响应报文

响应状态码,响应头,响应体

响应报文第一行是响应状态码,包含了基本的状态信息与含义原语

状态码 类别 含义
1XX Informational（信息性状态码） 接收的请求正在处理
2XX Success（成功状态码） 请求正常处理完毕
3XX Redirection（重定向状态码） 需要进行附加操作以完成请求
4XX Client Error（客户端错误状态码） 服务器无法处理请求
5XX Server Error（服务器错误状态码） 服务器处理请求出错

常见的状态码

  • 200:成功请求
  • 400:bad request 请求报文错误
  • 401:未授权,请求头中需要带有必要的授权信息
  • 403:forbidden 请求被服务器拒绝
  • 404:资源不存在
  • 500:服务器内部错误
  • 503:服务器忙

2xx表示成功请求,3xx通常需要额外操作,4xx是客户端错误,5xx是服务器的错误,经常出现5xx说明服务端开发人员水平欠佳

浏览器HTTP请求经历的过程

1.本地占用一个端口,查看本地host文件是否指定了请求URL的IP地址,如果是则2,否则3

2.直接访问IP地址主机的80端口,并准备开始建立连接,随后6

3.本地没有配置则从本地DNS查询URL,若有记录则4,否则5

4.解析出IP地址,然后2

5.查询远程DNS,返回IP同时存入本地,方便其他用户,然后2

6.基于TCP的三次握手,与服务器的80端口进程建立可靠连接

7.发送请求数据到服务器,服务器解析成HTTP请求报文后处理

8.服务器封装处理后的响应报文发送到客户端

9.浏览器解析Response,完成渲染

短连接与长连接

当浏览器访问一个包含多张图片的 HTML 页面时，除了请求访问 HTML 页面资源，还会请求图片资源。如果每进行一次 HTTP 通信就要新建一个 TCP 连接，那么开销会很大。

长连接只需要建立一次 TCP 连接就能进行多次 HTTP 通信。

  • 从 HTTP/1.1 开始默认是长连接的，如果要断开连接，需要由客户端或者服务器端提出断开，使用 Connection : close；
  • 在 HTTP/1.1 之前默认是短连接的，如果需要使用长连接，则使用 Connection : Keep-Alive。
