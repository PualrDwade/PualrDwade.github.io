一. 运行时数据区

Java虚拟机执行字节码指令,并且通过划分良好的运行时数据区进行协作,从而为java带来生命

需要注意的是,运行时常量池是位于方法区内部的

划分成为思维导图,便于理解与记忆

二. 垃圾回收(GC)

垃圾收集器主要是针对于线程共享的区域(堆,方法区)进行工作。

垃圾收集主要有两部分的工作:

  1. 判断对象是否可以被回收
  2. 采用一定的策略对可回收对象进行回收操作

2.1 可回收性质判断

2.1.1 引用计数法

为对象添加一个引用计数器，当对象增加一个引用时计数器加 1，引用失效时计数器减 1。引用计数为 0 的对象可被回收。

在两个对象出现循环引用的情况下，此时引用计数器永远不为 0，导致无法对它们进行回收。正是因为循环引用的存在，因此 Java 虚拟机不使用引用计数算法。

在上述代码中,a与b引用的对象实例互相持有了对象的引用,因此当我们把对a对象与b对象的引用去除之后,由于两个对象还存在互相之间的引用,导致两个Test对象无法被回收。

2.1.2 可达性分析法

以 GC Roots 为起始点进行搜索，可达的对象都是存活的，不可达的对象可被回收。

Java 虚拟机使用该算法来判断对象是否可被回收，GC Roots 一般包含以下内容：

  • 虚拟机栈中局部变量表中引用的对象
  • 本地方法栈中 JNI 中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中的常量引用的对象

2.1.3 方法区的垃圾回收

方法区被称为永久代,回收频率比较低,主要是进行常量的回收以及对类的卸载

大量使用反射与动态代理技术会产生很多临时使用的类,这是就要求虚拟机具备较好的类卸载效率

类卸载的必要条件:

  • 该类所有的实例都已经被回收，此时堆中不存在该类的任何实例。
  • 加载该类的 ClassLoader 已经被回收。
  • 该类对应的 Class 对象没有在任何地方被引用，也就无法在任何地方通过反射访问该类方法。

简单来说:类的实例没了,类的加载器没了,类的Class对象任何引用都没了,这样才满足被回收的条件

2.2 引用类型与可回收判断关系

无论是通过引用计数算法判断对象的引用数量，还是通过可达性分析算法判断对象是否可达，判定对象是否可被回收都与引用有关。

Java 提供了四种强度不同的引用类型。

2.2.1 强引用

被强引用的对象不会被回收

2.2.2 软引用

软引用可以通过JDK提供的SoftReference类来创建,软引用在内存不足的时候会被回收

软应用是手动GC不掉的

2.2.3 弱引用

从名字就可以知道,与强引用是对应的

被弱引用关联的对象一定会被回收，也就是说它只能存活到下一次垃圾回收发生之前。

垃圾回收有专门的线程,有回收的周期,可以直接手动GC掉

证明我们的论证是正确的,弱引用会保持对象可访问,直到下一次自动|手动GC

2.2.4 虚引用

虚引用又称幻影引用或者幽灵引用

又称为幽灵引用或者幻影引用，一个对象是否有虚引用的存在，不会对其生存时间造成影响，也无法通过虚引用得到一个对象。

为一个对象设置虚引用的唯一目的是能在这个对象被回收时收到一个系统通知。

使用 PhantomReference来创建虚引用。

2.3 垃圾回收算法

以下介绍一些虚拟机常用的回收算法

2.3.1 标记-清除算法

标记清除算法很简单,判断可回收的时候打上标记,在回收的时候对标记位置进行回收即可

不足：

  • 标记和清除过程效率都不高；
  • 会产生大量不连续的内存碎片，导致无法给大对象分配内存。

2.3.2 标记-整理算法

让所有存活的对象都向一端移动，然后直接清理掉端边界以外的内存。

2.3.3 复制算法(主流)

将内存划分为大小相等的两块，每次只使用其中一块，当这一块内存用完了就将还存活的对象复制到另一块上面，然后再把使用过的内存空间进行一次清理。

主要不足是只使用了内存的一半。

现在的商业虚拟机都采用这种收集算法回收新生代，但是并不是划分为大小相等的两块，而是一块较大的 Eden 空间和两块较小的 Survivor 空间，每次使用 Eden 和其中一块 Survivor。在回收时，将 Eden 和 Survivor 中还存活着的对象全部复制到另一块 Survivor 上，最后清理 Eden 和使用过的那一块 Survivor。

HotSpot虚拟机的 Eden 和 Survivor 大小比例默认为 8:1，保证了内存的利用率达到 90%。如果每次回收有多于 10% 的对象存活，那么一块 Survivor 就不够用了，此时需要依赖于老年代进行空间分配担保，也就是借用老年代的空间存储放不下的对象。

###2.4 垃圾收集器

2.4.1 概览

以上是 HotSpot 虚拟机中的 7 个垃圾收集器，连线表示垃圾收集器可以配合使用。

  • 单线程与多线程：单线程指的是垃圾收集器只使用一个线程，而多线程使用多个线程；
  • 串行与并行：串行指的是垃圾收集器与用户程序交替执行，这意味着在执行垃圾收集的时候需要停顿用户程序；并行指的是垃圾收集器和用户程序同时执行。除了 CMS 和 G1 之外，其它垃圾收集器都是以串行的方式执行。

2.4.2 GC的停顿与CPU吞吐率

垃圾回收相同数量的垃圾对象:

  • 若回收频率越高,则回收的停顿越低,用户交互越强,要求新生代空间越小越好,相对应的则CPU吞吐率下降(要频繁GC)
  • 若频率越低,则回收的停顿时间较长,CPU吞吐率较高,不需要额外时间来控制回收

缩短停顿时间是以牺牲吞吐量和新生代空间来换取的：新生代空间变小，垃圾回收变得频繁，导致吞吐量下降

三. 内存分配与回收策略

3.1 Minor GC与Full GC

  • Minor GC：回收新生代，因为新生代对象存活时间很短，因此 Minor GC 会频繁执行，执行的速度一般也会比较快。
  • Full GC：回收老年代和新生代，老年代对象其存活时间长，因此 Full GC 很少执行，执行速度会比 Minor GC 慢很多。

3.2 内存分配策略

  • 对象优先在Eden分配:Eden满则触发Minor回收
  • 大对象直接进入老年代:大数组,长字符串…
  • 长期存活进入老年代:每次经历过一次GC,年龄+1,达到规定值则转移到老年代
  • 动态对象年龄判定:也不一定达到指定年龄,可以动态判断,找出一个年龄众数(大于一半),则超过此年龄也可以进入老年代
  • 空间分配担保:由于Minor GC采用复制算法,需要把Eden+Servival1中存活对象复制到Servival2中,若Servival2分区空间不足以放下所有对象,则需要判断老年代分配额外空间是否能够担保,若能够保证放入新对象,则直接进行Minor GC,否则担保失败,出发Full GC。

新生代的Minor GC执行时间短,适合生存时间短的对象,而Full GC则速度慢,回收存活比较久的,要频率尽可能低。总之:尽可能在新生代被回收,活得久再去老年代

3.3 Full GC触发条件

对于 Minor GC，其触发条件非常简单，当 Eden 空间满时，就将触发一次 Minor GC。而 Full GC 则相对复杂，有以下条件:

  • 调用System.gc():只是建议虚拟机执行 Full GC，但是虚拟机不一定真正去执行。不建议使用这种方式，而是让虚拟机管理内存。
  • 老年代空间不足:
    老年代空间不足的常见场景为前文所讲的大对象直接进入老年代、长期存活的对象进入老年代等。为了避免以上原因引起的 Full GC，应当尽量不要创建过大的对象以及数组。除此之外，可以通过 -Xmn 虚拟机参数调大新生代的大小，让对象尽量在新生代被回收掉，不进入老年代。还可以通过 -XX:MaxTenuringThreshold 调大对象进入老年代的年龄，让对象在新生代多存活一段时间。
  • 空间分配担保失败:如上3.2

四. 类加载机制

4.1 类加载的时机

类是在运行期间第一次使用时动态加载的，而不是一次性加载所有类。因为如果一次性加载，那么会占用很多的内存

使用一个类的静态常量字段不会触发类的初始化,因为静态常量在编译时期就会存入调用类的常量池中,而不会触发定义此常量的类

4.2 类加载过程

类加载包括:加载->连接(验证-准备-解析-初始化)->使用->卸载这几个步骤

4.2.1 加载

加载是类加载的一个子过程,不要混淆

加载过程主要是类的二进制文件与方法区进行联动,主要做了三件事情

  • 类的全限名->类的二进制字节流(.class文件)
  • 类的静态存储结构->方法区的运行时存储结构
  • 在堆中生成此类的类实例(Class对象),作为该类方法区中各种数据的访问入口

4.2.2 验证

验证阶段主要是确保Class字节流包含的信息等是合法的且不会危害虚拟机的安全

4.2.3 准备

类变量是被 static 修饰的变量，准备阶段为类变量分配内存并设置初始值，使用的是方法区的内存。

实例变量不会在这阶段分配内存，它会在对象实例化时随着对象一起被分配在堆中。应该注意到，实例化不是类加载的一个过程，类加载发生在所有实例化操作之前，并且类加载只进行一次，实例化可以进行多次。

初始值一般为 0 值，例如下面的类变量 value 被初始化为 0 而不是 123。

如果类变量是常量，那么它将初始化为表达式所定义的值而不是 0。例如下面的常量 value 被初始化为 123 而不是 0。

准备是为类变量赋零值,真正的值需要在后面的阶段再赋予

4.2.4 解析

将常量池的符号引用替换为直接引用的过程。

其中解析过程在某些情况下可以在初始化阶段之后再开始，这是为了支持 Java 的动态绑定。

整个过程是循序开始的,但是不是顺序工作的,也就是他们的周期可能是交错的!

五. 对象创建的顺序

  1. 检查在方法区中是否存在对象的符号引用,若没有,则通过类全限名得到类的二进制字节流,检查类是否存在,若不存在则抛出异常,否则进行类加载
  2. 类加载完成之后,从堆中划分出内存
  3. 初始化对象的字段属性等对象信息
  4. 将对象赋给引用变量,完成类的创建
